| 2026-05-10 | R4a spec missed two implicit prerequisites that surfaced at /code time: (a) `.gitignore` allow-list extension for template file types (.ts, .tsx, .py, .go, .toml, .mod, .gitkeep, etc.) plus subdirectory re-includes (`!scaffolding/**/`) — without these, scaffolded template content was silently gitignored; (b) `new-project.sh` `find` pattern initially missed `.gitignore` (no extension), so the go-service template's `.gitignore` placeholder didn't get substituted. | 2026-05-10 | R3 Testing should ship BEFORE R2 Monitoring. Original ROADMAP had R2 next (telemetry on workflow effectiveness, leveraging the R1.5 heartbeat). Reality after R4a: every dev-platform "gate fast" run is a Bash one-liner cobbled together in conversation — taxonomy, syntax, JSON validity, install round-trip, live verify, heartbeat smoke, scaffold smoke, edge cases. No `scripts/gate_fast.sh` exists. Without R3, every spec from here on (R5 migration, R4b VSCode, additional templates) ships with conversation-derived gate coverage. R3 *prevents* drift; R2 only adds *visibility* to drift. Prevention is higher-leverage than visibility when neither exists yet. | 2026-05-10 | Bash helpers that maintain state across subshells need a shared file or env-var mechanism. R3's first `gate_fast.sh` ran each test suite via `bash "${runner}"` (subshell), and each runner's `PASS_COUNT` / `FAIL_COUNT` lived in its own subshell scope — the orchestrator's totals were ALWAYS the 5 lift checks alone, never the 37 suite results underneath. Caught only because /code re-ran gate_fast.sh and saw "5 PASS" when 42 was expected. Fix: shared `_GATE_COUNTS_FILE` (mktemp), exported env var, assert.sh appends to it, orchestrator reads at end. Pattern: anything maintained "across subshells" via in-memory counters silently breaks; use a shared sink. | dev-platform | active |
| 2026-05-10 | Docs that promise behavior the code doesn't deliver violate the Honesty rule. R3 `tests/README.md` told future contributors "the orchestrator discovers new runners automatically by walking `tests/<suite>/`" — but the orchestrator had a hardcoded suite list (`for suite in hooks commands taxonomy install scaffold; do`). Adding `tests/foo/` wouldn't auto-discover; user would also need to edit gate_fast.sh. Caught by /review. Two fix shapes: tighten the README (match the code) or loosen the code (match the README). Picked the latter — auto-discovery is the better long-term contract. Pattern: when /review surfaces a doc-vs-impl mismatch, ALWAYS prefer fixing the implementation to match the doc's promise unless the promise was itself wrong. | dev-platform | → Captured in Honesty rule (existing) |
| 2026-05-11 | Specs that change a script's output contract MUST include a Change that updates the regression-test suite asserting that contract. v0.5 Phase 1 Change 2 migrated `hooks/post-tool-heartbeat.sh` from text format (`tool=Bash`) to JSONL, but the spec didn't list updating `tests/hooks/post-tool-heartbeat/run.sh` (the v0.4 regression suite) as a Change. The omission surfaced when gate_fast went red mid-/code with 4 FAILs on the heartbeat fixture suite — implicit, not explicit. Pattern: when a /plan Change rewrites the output format / API / event shape of an artifact that has a downstream test, the spec MUST also list a Change to update that test in lockstep. Going forward, /plan reviews check every output-changing Change for a paired test-update Change. | dev-platform | active |
| 2026-05-11 | Per-Spec-Phase branching strategy adopted for v0.5 onward. Each Spec Phase (typically 2–5 Changes) → one feature branch (`v0.5/phase-1-schema-storage` etc.) → one PR assigned to the matching GitHub Milestone → squash-merge to main. Reviewed in /review session 2026-05-11; alternatives considered: per-Change PRs (too granular, 14 PRs for v0.5), per-Roadmap-Phase PR (too large, ~250-line aggregator + 4 hooks + tests in one diff), stay on main (no review surface for 2–10 dev team). Per-Spec-Phase is the Goldilocks granularity. Tooling enforcement (required CI status check on the taxonomy + format) ships in v0.7 Team Enablement. | dev-platform | active |
| 2026-05-11 | When you introduce a new file type into a directory that's governed by glob-based loops, audit ALL consumers in the same /code session — don't ship the file and hope the loops pick it up. v0.5 Phase 2's /review refactor added `hooks/_emit_event.py` (first `*.py` ever under `hooks/`). Required parallel updates to 4 files: (a) `.gitignore` allow-list (`!hooks/*.py`), (b) `scripts/install.sh` (add `*.py` to the deploy glob), (c) `scripts/verify.sh` (add `*.py` to the symlink-check loop), (d) the hooks/ directory README (none exists for hooks yet but worth flagging). Missing any of these breaks deploy silently — install would skip the file, verify wouldn't catch the gap, hooks would fall back to degraded behavior. Pattern: any new top-level file extension OR any new file in a glob-managed directory triggers a 4-point consumer audit before commit. | dev-platform | → Rule in `dev/CLAUDE.md` |
| 2026-05-11 | When /review surfaces "inconsistent X across N files," the answer isn't always "align them." Sometimes the asymmetry is deliberate and the fix is to document the rationale where readers look. v0.5 Phase 2's /review flagged inconsistent fallback-emission strategies across 4 hooks (session-start + post-tool-heartbeat have fallback echo; user-prompt + pre-tool-use are silent). Each has a defensible reason (privacy for user-prompt; pairing semantics for pre-tool-use). The fix was a "Fallback asymmetry" section in `monitoring/README.md`, NOT alignment. Pattern: when /review's QUALITY findings call out inconsistency, first ask "is the asymmetry deliberate?" — if yes, document the rationale; if no, align. | dev-platform | active |
| 2026-05-11 | Synthetic test fixtures must use PAST timestamps. v0.5 Phase 3's /test built an end-to-end fixture covering every metric path — initially using timestamps like `2026-05-11T09:00:00` while current local time was ~05:00. The aggregator's window filter (`ts > until`, where `until = now`) silently dropped all 12 fixture events. Surfaced as "event_count = 0" in the first assertion. Fix: backdate fixture timestamps (e.g., `2026-05-10T...`) so they fall inside any reasonable `--period` window. Pattern: when writing time-bounded test fixtures, anchor timestamps to a known-past date (or compute relative to "now" minus an offset); never use today-but-future times. The aggregator's `--period all` does NOT mean "ignore the upper bound" — it means `since=datetime.min`, but `until` is always `now`. | dev-platform | active |
| 2026-05-11 | Spec-faithful implementation can ship inconsistencies inherited from the spec itself. v0.5 Phase 3's /review flagged two cross-metric naming/structural inconsistencies (`code_invocations` vs `count`; `events_per_project` inline vs as a metric_* helper) — BOTH of which were literally what the spec prescribed. The fixes were correct (align to `count`; extract the helper), but the deeper lesson is: when /code implements a spec verbatim, /review should still audit for consistency that the spec might have missed. /plan agents are not infallible. Pattern: /review's QUALITY pass looks at the implementation in its full repo context, not just against the spec. Findings that contradict the spec are valid when consistency with the broader codebase wins; flag them as "spec deviation worth absorbing into the next spec revision" rather than treating the spec as immutable. | dev-platform | active |

| 2026-05-11 | Consumer Audit rule (just promoted to `dev/CLAUDE.md`) fired TWICE within v0.6's same /code session — within hours of the rule landing. (1) `extensions/vscode/` was the first subdirectory of `extensions/` to contain real files; gitignore had `!extensions/**/*.json` and `!extensions/**/*.md` but was missing `!extensions/**/` subdir re-include (same pattern as scaffolding/monitoring/tests/). | 2026-05-11 | Mock-binary pattern for testability — when a script depends on an external command (`code`, `python`, `docker`), the round-trip tests need to exercise the command's behavior without touching the live system. v0.6 introduced `tests/vscode/fixtures/mock-bin/code` — an extension-less Bash script that mimics `code --list-extensions` and `code --install-extension` via a `MOCK_STATE_FILE` env var. The test runner sets `PATH="${MOCK_BIN}:${PATH}"` so the mock takes precedence over the real binary. Same pattern works for any sufficiently-narrow CLI surface. Prerequisite: the script under test must accept a `--file <path>` (or similar) override so it doesn't hardcode-write to the live tracked file. v0.6 added that flag to `sync-vscode.sh` as part of the /review fix. Pattern: when adding a CLI script that touches external state, build in a state-file override flag from the start so future mock-based testing is cheap. | dev-platform | active |
| 2026-05-11 | Per-Spec-Phase branching is the default, but small Roadmap Phases (1 Spec Phase fitting in <200 LOC diff) can ship as a single branch/PR. v0.6 had nominally 3 Spec Phases (Tracking / Deploy+Test / Wire-up) but only 6 Changes totaling ~250 LOC — splitting into 3 PRs would have been per-PR ceremony exceeding per-PR content. The spec explicitly called out the deviation: "Single feature branch ... single PR (no Per-Spec-Phase strategy needed)." Result: 1 PR merged cleanly. Pattern: per-Spec-Phase is appropriate when Spec Phases are loosely coupled OR each Phase exceeds ~150 LOC; otherwise a single branch is cheaper. Future /plan agents should call out the strategy choice in the spec's Design Philosophy or Implementation Order section so /code knows whether to create one branch or many. | dev-platform | active |
| 2026-05-11 | Negative-test fixtures must assert on the SPECIFIC failure cause, not just exit nonzero. v0.7 Phase 1's first cut of `tests/taxonomy/run.sh` asserted only `expected_exit==1` on `bad-roadmap-{sprint,rprefix}.md`. Each "bad" fixture contained 1 targeted invalid line plus 2 valid lines (e.g. `v0.1: Foundation`) — meaning a future regex regression that mismatched a *valid* line would still produce exit 1 and the test would pass for the wrong reason. /review caught it. Fix: `run_roadmap_fixture` gained an optional `expected_match` 4th arg; harness now captures stderr (`2>&1`) and greps for the specific killed-prefix substring (`Sprint K:`, `R7:`). Pattern: every negative test needs two assertions — (1) it failed, AND (2) it failed for the documented reason. Exit-code-only is "did something break"; substring assertion is "did THIS break". | dev-platform | active |
| 2026-05-11 | GitHub Actions reusable-workflow checkout MUST use `github.workflow_sha`, NOT `github.workflow_ref`. v0.7 Phase 2's spec prescribed `ref: ${{ github.workflow_ref \|\| 'main' }}` for the dev-platform self-checkout inside `taxonomy-check.yml`. Per [GitHub Actions docs](https://docs.github.com/en/actions/learn-github-actions/contexts), `github.workflow_ref` returns the FULL ref-path string (`owner/repo/.github/...@refs/tags/v0.7`), which `actions/checkout`'s `ref:` cannot resolve as a git ref — invocation would fail at first consumer adoption. `github.workflow_sha` is the commit SHA the calling consumer's pinned tag resolves to. /test caught the spec bug before any consumer hit it. Pattern: for reusable workflows (`workflow_call` trigger) that need to check out their own repo, always use `github.workflow_sha`; comment why (people read `workflow_ref` and assume it's the right knob). | dev-platform | active |
| 2026-05-11 | A /review fix can introduce a new bug in a different context — verify portability before pinning a fix. v0.7 Phase 1's /review changed `Standard: $(dirname "$PROJECT_ROOT")/dev/CLAUDE.md` (wrong from temp-dir test harness, evaluated to `/tmp/dev/CLAUDE.md`) to `Standard: /home/rich/dev/CLAUDE.md` (correct on Rich's machine). Phase 2's /review caught that the hardcoded path is wrong on a CI runner — the path doesn't exist anywhere except Rich's box. Final fix: a public URL (`https://github.com/teelr/dev-platform/blob/main/CLAUDE.md`) — works on every machine, no localhost assumption. Pattern: when a /review fix changes how the code surfaces a system path / hostname / env-specific reference, audit the fix against *every* invocation context (local dev, test harness, CI runner, consumer adoption). Hardcoding to fix one context can break another. URLs > paths when the artifact is shared across machines. | dev-platform | active |
| 2026-05-11 | Fixture subdirectories under `tests/<suite>/fixtures/` MUST avoid both (a) directory names that collide with repo-wide gitignore patterns and (b) runnable `.sh` files that the gate orchestrator could pick up as test runners. v0.8 Phase 1 hit BOTH simultaneously: spec said `tests/fleet-gate/fixtures/projects/{pass-1,...}/gate.sh`, but `.gitignore:132 projects/` is an unanchored exclude (matches `projects/` anywhere in the tree, beats the earlier `!tests/**/*.sh` re-include by rule-order); AND `scripts/gate_fast.sh`'s find walked `tests/<suite>/**/*.sh` and would have executed each mock gate as a test runner (one would fail, one would sleep 999s and hang the entire gate). Fixes: (a) rename to `mock-projects/` to dodge the gitignore; (b) extend the gate's find with `! -path "*/fixtures/*"` codifying the contract "test runners live at `tests/<suite>/*.sh` or `tests/<suite>/<test>/*.sh`, NEVER under `fixtures/`". Pattern: when designing fixture trees, (1) `git check-ignore -v` every new file before treating it as tracked, (2) ensure executable `.sh` files inside `fixtures/` are explicitly excluded by the orchestrator's auto-discovery. Both v0.6's `mock-bin/<binary>` (extension-less) and v0.8's `mock-projects/<X>/gate.sh` are valid fixture shapes ONLY because the find pattern excludes them. | dev-platform | active |
| 2026-05-11 | When introducing a new `gh` (or any third-party CLI) invocation, smoke-test against the LOCALLY-INSTALLED CLI version before shipping. Don't trust docs that describe flags added in later releases. v0.8's `/merge` slash command spec used `gh pr checks --json name,bucket,state` — that `--json` flag was added in gh ~2.50; local gh is 2.45.0 and rejects it with "unknown flag: --json". Same root cause silently broke the session's earlier Monitor invocations (`gh pr checks N --json name,bucket 2>/dev/null \|\| echo '[]'` swallowed the error and returned `[]` every poll, so PR-CI watchers timed out instead of detecting completion). Fix: switched to `gh pr view <N> --json statusCheckRollup,mergeable,mergeStateStatus` which works on every gh version the workflow supports. Pattern: every new gh subcommand goes through `gh <subcommand> --help` on the LOCAL install first to confirm flag availability — version-skewed docs are a real hazard. Same shape applies to docker, kubectl, jq, awk — when the OS package version may lag the upstream docs by months. | dev-platform | active |
| 2026-05-11 | /review's "dead code" / "unused branch" / "redundant" claims MUST be verified against the production data the code actually consumes, not just by reading the file in isolation. v0.8 Phase 3's /review flagged the relative-path branch in `fleet-install-template.sh:117-121` as dead defensive code based on the (wrong) assumption that the registry uses absolute paths. The proposed fix — reject relative paths with an explicit error — would have broken every entry in `monitoring/projects.json` (atlas, kermit, kermit-pa, keystone all use relative `projects/<name>`; only `dev-platform` uses `.`). The `fix all` smoke check (`./scripts/fleet-install-template.sh --project atlas`) caught the regression on the first invocation; reverted before re-test. Pattern: before flagging a branch as dead, run `jq` / `grep` / `git log` against the actual data sources the code reads — even a single one-liner suffices. This is the existing "Verify Against Source of Truth" rule firing on /review's side, not implementation-side. /review's QUALITY assertions deserve the same verification rigor as /code's claims. | dev-platform | active |
| 2026-05-11 | When a script has N modes (dry-run/apply) × M state combinations (target-exists / target-absent / disabled / missing-prereq), the test suite must exercise the CROSS PRODUCT, not just each axis once. v0.8 Phase 3's first cut had assertions for: dry-run with absent target (Check 6), --apply with absent (Check 7), --apply with existing (Check 8). All three PASSed and /test reported PASS. /review caught a real BUG: dry-run with existing target refused early (because the refuse-to-clobber guard didn't gate on `APPLY=1`), suppressing the entire dry-run plan. The (dry-run × existing) cell was simply missing from the matrix; each axis was covered but their intersection wasn't. Fix landed as test Check 14 + the implementation fix. Pattern: when /plan writes acceptance tests for a mode-driven script, enumerate the cross product table (rows = modes, columns = preconditions) and require every cell to have at least one assertion. Single-axis coverage gives false confidence — the bug lives in the off-diagonal cells. | dev-platform | active |
