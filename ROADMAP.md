# dev-platform Roadmap

Spec sequence for the `teelr/dev-platform` repo. Each entry is a Roadmap Phase — its detailed Spec lives under `tasks/`, and each maps to a GitHub Milestone in the repo with the same `v<MAJOR>.<MINOR>: <Title>` name.

**Version semantics:** `v0.x` = pre-1.0 build-out (the dev-platform standards + tooling are still mutable). `v1.0` cuts when v0.1–v0.9 all ship and dev-platform is feature-complete for both solo and 2–10 dev teams. Subsequent enhancements become `v1.1`, `v1.2`, …; breaking workflow changes become `v2.0`. Projects that vendor dev-platform can pin to a specific version for stability.

- **v1.2: Pre-commit Git Hook** *(complete — 2026-05-19, `tasks/dev-platform-pre-commit-hook-spec.md`)* — universal opt-in pre-commit hook converting the /gate-fast-before-commit rule from documentation to mechanical enforcement. `shell/git-hooks/pre-commit` looks for `scripts/gate_fast.sh` at the repo root, no-ops where absent, refuses commits on FAIL otherwise; bypass via `SKIP_GATE_FAST=1` env var for WIP commits on private branches. - **v1.4: Concurrent Dev Isolation** *(complete — 2026-06-23, `tasks/dev-platform-concurrent-dev-isolation-spec.md`)* — lets more than one Claude Code chat run against the same project without tripping over each other. Two layers: (1) **files and branch** — `/code` starts opted-in projects in their own `.claude/worktrees/<branch>` via the harness `EnterWorktree` tool (instead of `git checkout -b` in a shared tree), with `shell/worktree/link-deps.sh` symlinking the project's heavy git-ignored paths (`.env`, `node_modules`) in per a committed `.claude/worktree-deps` manifest; `/merge` tears the worktree down with `ExitWorktree` + `git worktree remove`. (2) **shared app/database** — chats take turns: `shell/worktree/gate-lock.sh` provides a `with_gate_lock` flock helper (lockfile in the shared git common dir) so concurrent gates serialize the backend stop/restart instead of fighting over the port. dev-platform's own `gate_fast.sh` adopts the lock around its live `~/.claude/` verify as the reference. New `worktree` install category; opt-in per project via the marker file; default (no marker) keeps `git checkout -b` unchanged. Does NOT provide per-chat live apps (per-chat ports + databases) — explicitly out of scope. Plain-English how-to + per-project adoption runbook in `docs/CONCURRENT-DEV.md`. Gate at 172 PASS (+9 from `tests/worktree/`). Post-merge: per-project adoption (Kermit, Kermit PA, Keystone, SQRL) runs from each project's own session; `sync-milestones.sh --apply`; `gh release create v1.4`; v1.4 Milestone closed; consumer-template default-pin bumped `@v1.3` → `@v1.4` (chore PR after tag exists).
- **v1.6: Local Settings Isolation** *(complete — 2026-06-28, `tasks/dev-platform-local-settings-isolation-spec.md`)* — stops Claude Code's "always allow" permission grants from polluting the tracked (public) repo. Root cause: `~/.claude/settings.json` was a symlink into `settings/settings.json`, so every runtime grant wrote straight through into version control (months of recurring drift). The Claude Code docs don't document where grants are written or how to redirect them (claude-code-guide research, all "NOT DOCUMENTED"), so the fix is structural: `settings.json` is now **merge-deployed as a real local file** (not a symlink) via `scripts/merge_settings.py` (unions the baseline's permission arrays into the live `~/.claude/settings.json`, baseline wins on config keys) — grants accumulate locally and never reach the repo; `settings.local.json` is seeded once from a secret-free `settings/settings.local.json.example`. `verify.sh` gains a real-file-superset check (and flags a regression if `settings.json` is ever a repo symlink again); `claude-global.md`/`keybindings.json` stay symlinked (not runtime-writable). **Scope honesty:** stops *future* pollution; does NOT prune the existing 113-entry baseline; the union-merge can add but not retract a permission. Surfaced (but did not fully remediate) that `settings/settings.local.json` carried plaintext DB passwords — now replaced by a secret-free `.example` + local-only live file. No post-merge step.

Ships the parts that live in dev-platform: (1) the **outbound standard** in `docs/CROSS-REPO-COMMS.md` — the dependency announces versions via GitHub Releases + release notes, consumers watch the repo / run Dependabot/Renovate, the `HARNESS_REPLIES_INBOX.md` broadcast-doc relay is deprecated; (2) a copy-paste **Dependabot consumer template** at `extensions/github-actions/dependabot-consumer-template.yml`; (3) `monitoring/comms-consumers.json` (consumer → upstream repo / label registry) + `scripts/setup-consumer-labels.sh` (idempotent `gh label create`, dry-run default) for the `consumer:*` triage labels; (4) the **delivery checker** `scripts/check-comms-delivery.sh` + `monitoring/comms_delivery.py` — scans every post-2026-06-28 `communique-to-<dep>-*` file and FAILs if it doesn't link a live upstream issue (the 2026-06-28 "filed but never delivered" failure mode). The checker is standalone (gh network calls, maybe-uncloned paths), not in `gate_fast`; its offline mock-`gh` suite is. **Scope honesty:** dev-platform ships the standard + template + tools; the harness actually cutting Releases and each consumer enabling Dependabot are post-merge per-repo coordination, not performed here. Gate at 190 PASS (+18 from `tests/comms-labels/` + `tests/comms-delivery/`). Post-merge: `setup-consumer-labels.sh --apply` against `teelr/kermit-harness`; harness stops relaying broadcast docs and cuts Releases from its own session; consumers adopt the Dependabot template from their own sessions; `sync-milestones.sh --apply`; `gh release create v1.5`; v1.5 Milestone closed; close issue #34.
- **v1.3: Mandatory Review Gate** *(complete — 2026-06-07, `tasks/dev-platform-mandatory-review-spec.md`)* — promotes `/review` from optional ("for risky/large changes") to a **mandatory** step in the canonical chain, between `/code` and `/gate fast`; new chain: `/plan → /code → /review → /gate fast → commit → push → /pr → CI → /merge → post-merge`. Second half (option 3): `/code` gains an explicit "Adversarial Self-Review" step (re-read your own diff as a hostile reviewer before reporting), distinct from the independent `/review` gate. Lockstep updates across the authoritative contract (`CLAUDE.md`, `settings/claude-global.md`), commands (`commands/code.md`, `commands/review.md`), published docs (`docs/index.md`, `skills/WORKFLOW_MANUAL.md`), the three scaffolding READMEs, and the migration tooling (`migrate-workflow-chain.sh` + `audit-project-drift.sh` now detect the review-less chain via `/code → /gate` and insert `/review`). Gate at 163 PASS (+5 from new `tests/migration/` assertions). Critical hazard navigated: the audit greps `CLAUDE.md` for `/code → /gate`, so no bare review-less chain string may appear in dev-platform's own `CLAUDE.md` (else self-flag) — verified `audit --project dev-platform` reports CLEAN. Post-merge: re-run `migrate-workflow-chain.sh --apply` against each consumer project to insert `/review` into their chain line; `sync-milestones.sh --apply`; `gh release create v1.3`; v1.3 Milestone closed; consumer-template default-pin bumped `@v1.2` → `@v1.3` (chore PR after tag exists).
- **v1.7: Plan-Time Isolation** *(in progress — 2026-07-01)* — moves branch/worktree creation from `/code` time to `/plan` time: `/plan` now derives the feature slug from `$ARGUMENTS`, creates the branch (`git checkout -b`) or worktree (`EnterWorktree`, for opted-in multi-chat projects) right when the user decides what to build, and renames the current tmux window to the slug if `$TMUX` is set — so two concurrent `/plan` sessions never collide writing `tasks/*.md` or `planning.md` on a shared branch. `/code`'s own branch-creation step becomes the fallback for hand-authored specs. Adds `shell/new-session.sh`, a personal CLI+tmux helper (`<project-dir> <branch-name> [window-name]`) for deliberately starting a second, fully parallel worktree + tmux window + `claude` session when the branch name is already known upfront. `/review` caught one real bug before commit: the worktree-mode instructions referenced capturing `MAIN=$(git rev-parse --show-toplevel)` only in a trailing note, after the point where `EnterWorktree` would have already re-rooted the session — fixed to capture `MAIN` as an explicit first action, matching `/code`'s existing ordering.
