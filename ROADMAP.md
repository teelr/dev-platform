# dev-platform Roadmap

Spec sequence for the `teelr/dev-platform` repo. Each entry is a Roadmap Phase — its detailed Spec lives under `tasks/`, and each maps to a GitHub Milestone in the repo with the same `v<MAJOR>.<MINOR>: <Title>` name.

**Version semantics:** `v0.x` = pre-1.0 build-out (the dev-platform standards + tooling are still mutable). `v1.0` cuts when v0.1–v0.9 all ship and dev-platform is feature-complete for both solo and 2–10 dev teams. Subsequent enhancements become `v1.1`, `v1.2`, …; breaking workflow changes become `v2.0`. Projects that vendor dev-platform can pin to a specific version for stability.

- **v0.1: Foundation** *(complete — 2026-05-08, `tasks/dev-platform-foundation-spec.md`)* — repo restructure, top-level dir contracts, install / uninstall / verify scripts. Migrated `~/.claude/{commands,skills/WORKFLOW_MANUAL.md,settings.json}` into the repo as the source of truth; live cutover executed against `~/.claude/`.
- **v0.2: Global Claude + Hooks Coverage** *(complete — 2026-05-09, `tasks/dev-platform-global-claude-and-hooks-spec.md`)* — closes the gateway-policy gap: tracks `~/.claude/CLAUDE.md` as `dev/settings/claude-global.md`, ships first hook script (`hooks/post-tool-heartbeat.sh`) plus the `settings.json` hooks-block wiring. 13 symlinks now flow from `dev/` → `~/.claude/`. The heartbeat is the data source v0.5 will consume.
- **v0.3: Project Scaffolding** *(complete — 2026-05-10, `tasks/dev-platform-r4a-scaffolding-spec.md`)* — three starter templates (`go-service`, `python-agent`, `next-frontend`) under `scaffolding/`, orchestrator at `scripts/new-project.sh`, conversational Q&A pattern at `docs/NEW-PROJECT.md`, Scope-rule scaffolding carve-out inline in `dev/CLAUDE.md`. Assistant runs the scaffold; user describes the project. v0.6 (VSCode user-profile + extensions list) deferred to its own spec.
- **v0.4: Testing** *(complete — 2026-05-10, `tasks/dev-platform-r3-testing-spec.md`)* — `scripts/gate_fast.sh` orchestrator + 5 auto-discovered test suites under `tests/` (hook fixtures, command frontmatter validator, taxonomy self-test, install round-trip, scaffold smoke). 42 mechanical checks aggregate to a single PASS/FAIL in < 1s. Replaces conversation-derived gate-fast with a runnable script every spec from here on rides. Auto-discovery: any new `tests/<suite>/*.sh` is picked up without editing the orchestrator. Deferred to future small specs: `gate_full.sh` (per-template builds), pre-commit git hook, performance benchmarks.
- **v0.5: Monitoring** *(IN FLIGHT — Phase 1/4 implemented 2026-05-11, `tasks/dev-platform-r2-monitoring-spec.md`)* — telemetry aggregation on top of the v0.2 heartbeat data: gate pass rate, `/code` retry counts, `/review` catch rate, hook execution time per project. Schemas in `monitoring/`, additional collectors as hook scripts in `hooks/`. Per-Spec-Phase branching strategy: 4 feature branches → 4 PRs, all assigned to GitHub Milestone `v0.5: Monitoring`. **Phase 1 (Schema + Storage)** on branch `v0.5/phase-1-schema-storage` — schema + JSONL heartbeat migration + gitignore allow-list. **Phase 2 (Collectors)** next — SessionStart / UserPromptSubmit / PreToolUse hooks + gate_fast self-instrumentation. v0.4 was promoted ahead of v0.5 to land first because every spec needs gate coverage to ship safely; that's now in place, so v0.5 telemetry is the next natural extension.
- **v0.6: VSCode Coverage** *(planned)* — VSCode user-profile config (settings.json, keybindings, snippets), statusline scripts, tracked `extensions/vscode/extensions.json` extensions list with sync helper. Deferred from v0.3 (the scaffolding spec) because the remote-SSH client/server split deserves its own design pass.
- **v0.7: Team Enablement** *(planned)* — scale the dev-platform from solo to a 2–10 dev team on GitHub: (1) extend `check_spec_taxonomy.sh` to scan `ROADMAP.md` + `planning.md` for non-conforming Roadmap Phase headers (must match `v<MAJOR>.<MINOR>:`), (2) GitHub Actions workflow template at `extensions/github-actions/dev-platform-gate.yml` deployed into each project's `.github/workflows/` via `install.sh` as a required status check, (3) PR bot step in the same workflow that auto-comments on taxonomy violations with a link to the canonical doc, (4) GitHub Pages render of `dev-platform/docs/` so the team has one canonical URL for the standard, (5) GitHub Milestones automation that mirrors `ROADMAP.md` entries to repo Milestones (idempotent — re-running updates rather than duplicating). Tracker sync (Milestones → external trackers like Linear or Jira) deferred until one is chosen. v0.7 is the team-scale equivalent of what v0.4 was for solo: the enforcement gate everything else depends on; v0.8 (cross-project orchestration) and v0.9 (migrations) both assume v0.7's CI enforcement is in place.
- **v0.8: Cross-project orchestration** *(planned)* — fleet-level operations across every project under `projects/`: run `gate_fast` across all projects, surface a status dashboard (gate pass-rate, drift count, last-commit recency per project), automated drift correction, release/version management for the platform itself. v0.5 gives single-project visibility; v0.8 aggregates across the fleet. Deliberate scope expansion beyond "manage one dev environment" — pre-v0.8 the Scope rule barred project-touching work; v0.8 walks that back for *read-only* and *opt-in correction* operations only (still no per-project feature work from here). Depends on v0.7 — without CI enforcement, cross-project dashboards surface drift they can't prevent.
- **v0.9: Migration tooling** *(planned)* — auto-migrate older project layouts (legacy taxonomy, scattered config) onto the standard project structure described in `CLAUDE.md`. Includes the cross-project renaming pass that the v0.7 enforcement rule requires: `keystone` Sprint K/L/M → `v<MAJOR>.<MINOR>`, `kermit` + `kermit-pa` bare `Phase N` → `v<MAJOR>.<MINOR>`. Also handles legacy `R<N>`-prefixed spec filenames in dev-platform itself (`tasks/dev-platform-r2-*.md` → `tasks/dev-platform-monitoring-spec.md`, etc. — a clean-up deferred to v0.9 to avoid mid-flight churn). Depends on v0.7+v0.8 — migrations need the enforcement layer in place first so they actually stick.
- **v1.0: Feature-complete for solo + team** *(planned)* — cuts the first stable release tag once v0.1–v0.9 ship. dev-platform is then considered production-ready for both solo developers and 2–10 dev teams. Subsequent enhancements become `v1.1`, `v1.2`, …; breaking workflow changes become `v2.0`.

Canonical roadmap detail (per-Roadmap-Phase exit criteria, dependencies, demo definition) lives in `tasks/dev-platform-roadmap.md` — created when v0.5 starts.
